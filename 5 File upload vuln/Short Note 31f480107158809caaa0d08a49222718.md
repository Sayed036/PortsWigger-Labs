# Short Note

## 🔷 What Are File Upload Vulnerabilities?

- A web server **lets users upload files** without properly checking the file's **name, type, contents, or size**.
- Even a simple image upload can be abused to upload **dangerous files** like server-side scripts.
- Some attacks work just by **uploading the file**. Others need a follow-up HTTP request to **trigger execution**.

---

## 💥 Impact

Depends on two things:

1. **What the server fails to validate** (type, size, contents, name)
2. **What happens to the file after upload**

**Worst case scenarios:**

- Server allows `.php` / `.jsp` files → attacker uploads a **web shell** → full server control
- Filename not validated → attacker **overwrites critical files**
- Combined with directory traversal → file uploaded to **unexpected locations**
- File size not checked → **DoS attack** (fills up disk space)

---

## ⚙️ How Do These Vulnerabilities Arise?

- Developers try to add validation, but it's often **flawed or bypassable**
- Common mistakes:
    - Blacklisting dangerous extensions (but missing some)
    - Trusting the `Content-Type` header (easily faked with Burp)
    - Inconsistent validation across different directories/servers

---

## 🌐 How Web Servers Handle Static Files

When a request comes in:

1. Server reads the **file extension**
2. Matches it to a **MIME type** using a config map
3. Then one of three things happens:

| File Type | What Happens |
| --- | --- |
| Non-executable (image, HTML) | Sent directly to client |
| Executable + server configured to run it (e.g. PHP) | **Server runs the script**, sends output |
| Executable but server NOT configured to run it | Error, or file served as **plain text** (can leak source code!) |

> 💡 **Tip:** Check `Content-Type` response header — it hints at what the server thinks the file is.
>

## 🐚 Exploiting Unrestricted Uploads → Web Shell

If a server allows server-side scripts (PHP, Python, Java) and runs them:

**Simple file read shell:**

php

`<?php echo file_get_contents('/path/to/target/file'); ?>`

**Full command execution shell:**

php

`<?php echo system($_GET['command']); ?>
```
Then access it like:
```
GET /uploads/exploit.php?command=id
```

A working web shell = **full server control** (read/write files, exfiltrate data, pivot to internal network).

## 🧪 Exploiting Flawed Validation

### 1️⃣ Flawed File Type Validation (Content-Type Bypass)

- When uploading, the browser sends `Content-Type: image/jpeg`
- Some servers **only check this header**, not the actual file contents
- Fix: just **change the Content-Type header in Burp** to `image/jpeg` while sending a `.php` file → server accepts it

### 2️⃣ Preventing Execution in Upload Directories

- Servers may block execution in upload folders → your PHP file is returned as **plain text**, not executed
- **Trick:** Try uploading the file to a **different directory** that has less strict controls (using directory traversal in the filename)

### 3️⃣ Insufficient Blacklisting of Extensions

- Blocking `.php` doesn't block everything. Try alternative extensions:
  - `.php5`, `.php7`, `.phtml`, `.shtml`, `.phar`

#### Overriding Server Config Files

- **Apache:** Upload a malicious `.htaccess` file to a directory:
```
  AddType application/x-httpd-php .jpg
```
  Now `.jpg` files get **executed as PHP**!

- **IIS:** Upload a malicious `web.config` file with custom MIME mappings

#### Obfuscating File Extensions

Many tricks to bypass extension checks:

| Technique | Example |
|---|---|
| Change case | `exploit.pHp` |
| Multiple extensions | `exploit.php.jpg` |
| Trailing dot | `exploit.php.` |
| URL encode the dot | `exploit%2Ephp` |
| Null byte injection | `exploit.php%00.jpg` |
| Semicolon trick | `exploit.asp;.jpg` |
| Unicode tricks | `xC0 x2E` → decoded as `.` |
| Nested extension bypass | `exploit.p.phphp` (removing `.php` still leaves `.php`) |

### 4️⃣ Flawed File Content Validation

- Server checks **file content** (not just extension) — e.g., checks if image has dimensions
- **Bypass:** Use **ExifTool** to create a **polyglot file** — a valid JPEG image that also contains PHP code in its metadata
```
  exiftool -Comment="<?php echo system(\$_GET['cmd']); ?>" image.jpg -o exploit.php.jpg`

- JPEG magic bytes: `FF D8 FF` — keep these at the start, add payload inside

### 5️⃣ Race Condition Exploits

- Some servers: upload → store temporarily → validate → delete if bad
- **Attack:** Between upload and deletion, the file exists briefly → **send a request to execute it immediately**
- Very hard to detect in blackbox testing (need source code leak)

### Race Conditions in URL-based Uploads

- Server fetches file from a URL, stores temporarily, then validates
- If temp directory name uses weak random (like PHP's `uniqid()`), it can be **brute-forced**
- **Trick:** Upload a large file to extend processing time → bigger window to brute-force the temp path

## 🌀 Exploiting Without Remote Code Execution

### Stored XSS via Upload

- Upload an **HTML or SVG file** containing:

html

  `<script>alert(document.cookie)</script>
```
- If the file is served to other users → **Stored XSS**
- Only works if file is served from the **same origin**

### XXE via File Parsing

- If server parses XML-based files (`.docx`, `.xlsx`, `.svg`)
- Craft a malicious file with **XXE payload** → server processes it → reads internal files or makes internal requests

---

## 📤 Uploading via HTTP PUT Method

Some servers support `PUT` requests — can upload files without any upload form:
```
PUT /images/exploit.php HTTP/1.1
Host: vulnerable-website.com
Content-Type: application/x-httpd-php

<?php echo file_get_contents('/etc/passwd'); ?>
```

> 💡 **Tip:** Send `OPTIONS` request to an endpoint — if response includes `PUT` in `Allow` header, the method is supported.

---

## 🛡️ How to Prevent File Upload Vulnerabilities

| Prevention | Details |
|---|---|
| ✅ Use **whitelist** of allowed extensions | Easier to define safe types than block all bad ones |
| ✅ Validate filename | No `../` or traversal sequences |
| ✅ Rename uploaded files | Avoid overwriting existing files |
| ✅ Don't move to permanent storage until validated | Validate in temp/sandbox first |
| ✅ Use established frameworks | Don't write your own upload validation |`
